SaaS Infrastructure Orchestrator — Production Architecture v1.0
--------------------------------------------------------------------------
1. 🎯 Назначение системы
--------------------------------------------------------------------------
Оркестратор — это control plane распределённой инфраструктуры, который:
	- принимает события об оплате
	- принимает события о подписках
 	- управляет VPN и Storage ресурсами
	- распределяет пользователей по нодам
	- обеспечивает отказоустойчивость
	- синхронизируется с биллингом
--------------------------------------------------------------------------
++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------
2. 🧩 Компоненты системы
--------------------------------------------------------------------------
2.1 Billing Provider
--------------------------------------------------------------------------
Отвечает за:
	- регистрацию пользователей
	- тарифы (plans)
	- оплату
	- подписки
--------------------------------------------------------------------------
📌 Source of truth:
--------------------------------------------------------------------------
	- пользователи	
	- платежи
	- подписки
--------------------------------------------------------------------------
Billing decision:
--------------------------------------------------------------------------
Option A:
	- FossBilling + custom YooKassa payment adapter
	- YooKassa не считается готовым out-of-the-box gateway для v1
	- потребуется свой payment gateway / webhook integration
	- custom YooKassa adapter является частью billing side, а не core Orchestrator

Option B:
	- BILLmanager или другой billing provider с готовой YooKassa интеграцией

📌 Архитектура Orchestrator НЕ должна зависеть от конкретного billing продукта.
Все вызовы идут через BillingProvider interface.
--------------------------------------------------------------------------
2.2 Orchestrator (Core)
--------------------------------------------------------------------------
Отвечает за:
	- webhook обработку
	- очередь задач
	- provisioning
	- балансировку
	- управление нодами
	- хранение состояния
	- Admin UI
--------------------------------------------------------------------------
📌 Source of truth:
--------------------------------------------------------------------------
	- инфраструктура
	- provisioning
--------------------------------------------------------------------------
2.3 VPN Nodes (3x-ui)
--------------------------------------------------------------------------
	- 5–20 нод
	- создают VPN пользователей
	- возвращают subscription link
	- управляются через API
	- Orchestrator управляет 3x-ui удалённо, напрямую через API панели
	- на VPN ноды не устанавливается отдельный агент / adapter service
--------------------------------------------------------------------------
2.4 S3-compatible Storage Backend
--------------------------------------------------------------------------
	- Orchestrator не реализует raw storage сам
	- создаёт bucket / access / policy / quota через StorageProvider interface
	- default v1 backend: MinIO
	- alternative backend: Garage для распределённого self-hosted storage
	- future option: SeaweedFS
	- Storage backend должен быть S3-compatible
--------------------------------------------------------------------------
Provider decision:
--------------------------------------------------------------------------
Storage Provider фиксируется интерфейсом, а не конкретной реализацией.
Для v1 рекомендуется MinIO как основной backend.
Garage допускается как альтернатива, если нужна распределённость между VPS / локациями.

4. 🧠 Основная логика
--------------------------------------------------------------------------
4.1 Webhook
--------------------------------------------------------------------------
принимает события:
	- payment_paid
	- subscription_cancel
	- subscription_expired
	- plan_changed
	- не содержит логики
	- валидирует API key
	- валидирует подпись
	- проверяет idempotency key события
	- отправляет задачу в очередь
--------------------------------------------------------------------------
4.2 Queue (Bull + Redis)
--------------------------------------------------------------------------
	- асинхронная обработка
	- retry механизм
	- устойчивость к сбоям
--------------------------------------------------------------------------
4.3 Worker (Core Engine)
--------------------------------------------------------------------------
Flow:
	1. Получить job
	2. Проверить валидность данных
	3. Проверить идемпотентность через processed_events
	4. Найти существующий Provision по externalSubscriptionId
	5. Если есть → использовать его (sticky)
	6. Если нет → создать новый
	7. Выбрать ноды
	8. Создать ресурсы
	9. Обновить БД
	10. Записать processed event / job результат
	11. Отправить результат в Billing Provider
--------------------------------------------------------------------------
++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------

6. ⚖️ Балансировка нагрузки
--------------------------------------------------------------------------
Алгоритм:
	Least Loaded
Правила:
	- только isActive = true
	- только status = active
	- currentLoad < capacity
	- выбираем минимальную загрузку
	- финальное увеличение currentLoad выполняется атомарно с проверкой capacity
	- DB transaction не держится во время внешних API вызовов
--------------------------------------------------------------------------
++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------
6.1 🎛 Plan-driven VPN limits
--------------------------------------------------------------------------
При создании / обновлении VPN клиента:
	- взять maxDevices из Plan
	- передать maxDevices в 3x-ui client settings как limitIp
	- если maxDevices не задан → использовать 0 (без IP limit) или дефолт тарифа

Пример:
	Plan.maxDevices = 3
	3x-ui client.limitIp = 3

📌 Это ограничивает количество одновременных IP / устройств
на уровне VPN панели, а не только внутри Orchestrator.

7. 🧠 Sticky Allocation (ВАЖНО)
--------------------------------------------------------------------------
Если Provision уже существует:
	→ всегда использовать те же ноды
📌 предотвращает миграции и баги
📌 автоматическая миграция между нодами не входит в v1,
но модель должна хранить состояние нод так, чтобы migration flow можно было добавить в v2.

9. 🔐 Idempotency
--------------------------------------------------------------------------
Webhook может приходить несколько раз.
Используем:
	- processed_events.event_id (unique)
Правила:
	- не создавать дубликаты
	- eventId обязателен для всех webhook events
	- Provision уникален по externalSubscriptionId, а не по externalPaymentId
	- externalPaymentId относится к платежному событию, а не к жизненному циклу Provision
	- проверять статус перед выполнением

++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------
10. 🔁 Retry Strategy
--------------------------------------------------------------------------
attempts: 3
backoff:
  1 → 5s
  2 → 15s
  3 → 30s
После:
	- status = failed
	- ручной retry из UI
--------------------------------------------------------------------------
++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------
11. ❌ Обработка ошибок
--------------------------------------------------------------------------
При ошибке:
	- выполнить rollback частично созданных ресурсов
	- уменьшить currentLoad только если он уже был увеличен после успешного создания ресурса
	- сохранить error
	- статус → failed
--------------------------------------------------------------------------
++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------
12. 🔄 Deprovisioning
--------------------------------------------------------------------------
Триггеры:
	- cancel
	- expired
Действия:
	- отключить / удалить VPN клиента
	- storage НЕ удалять сразу
	- заморозить storage
	- storageStatus = frozen
	- status = suspended или cancelled
	- уменьшить load активных ресурсов
	- физическое удаление storage только отдельным cleanup job после deleteAfter

++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++
--------------------------------------------------------------------------
13. ❤️ Health Monitoring
Периодическая проверка:
	- доступность ноды
	- latency
	- ошибки API
Если проблема:
	- isActive = false
	- status = inactive
Если нода заблокирована провайдером / недоступна пользователям:
	- status = blocked
	- новые пользователи на ноду не назначаются
	- существующие Provision остаются привязанными к ноде до ручной или v2 миграции

13.1 🔁 Node Migration (v2 placeholder)
--------------------------------------------------------------------------
Цель:
	- перенести пользователей с проблемной VPN ноды на новую рабочую ноду

Причины:
	- нода заблокирована
	- нода удаляется из инфраструктуры
	- долгий downtime
	- ручное решение администратора

v1:
	- автоматическую миграцию НЕ выполняем
	- Admin UI показывает affected provisions для blocked/draining ноды
	- ручное изменение vpnNodeId не является штатным flow v1
	- администратор принимает решение о ручной поддержке пользователя вне автоматической миграции

v2:
	- Migration job
	- создание VPN клиента на новой ноде
	- обновление subscriptionLink в Billing Provider
	- удаление старого клиента, если старая нода доступна
	- аудит миграции
